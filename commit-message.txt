Fix read-state persistence, cache/performance issues, security flaws,
folder settings, and add content prefetching

READ STATE
- Root cause of "some articles show as read and it's inconsistent
  which": getArticleState did a plain select('*') with no .range() or
  .order(). Supabase/PostgREST caps any single response at 1000 rows
  by default -- with article_state past 2,000 rows, every load saw an
  incomplete AND unstably-ordered subset (no ORDER BY means no
  guarantee which rows come back first, especially on a table this
  frequently written to). Fixed with proper .range() pagination and
  .order('key') so it's deterministic. Added article_state.updated_at
  (trigger-set) so this class of bug is actually diagnosable in the
  future instead of requiring guesswork.

PERFORMANCE
- article_cache full-table scans: the startup cache-prune and the
  feed-rules PATCH route both pulled the ENTIRE article_cache table's
  content (12MB+, one row up to 771KB) into memory just to filter by
  feedId/timestamp in JS. Confirmed via Vercel logs as the direct cause
  of a "canceling statement due to statement timeout" error that took
  down a /api/feeds/fetch-all request with a 504, and of per-feed
  settings saves feeling slow/crashed. Replaced with single filtered
  SQL deletes (cacheDeleteByFeedId/s, cachePruneExpired, cacheClearAll)
  and added the feed_id/ts indexes needed to make them fast. Removed
  the now-dead full-table-pull method entirely rather than leave it
  for a future caller to reach for again.
- Folder-level settings PATCH busting cache with one query per feed in
  the folder -> single batched delete.
- resolveFeedRules was merging titleBlocklist/inlineBrowser on every
  feed on every fetch-all call for fields nothing server-side actually
  reads (confirmed against fetcher.js) -- trimmed to just what's used.
- Auto-refresh timer closed over a stale `feeds_` from whenever the
  effect last ran; added feedsRef so it always sees the current list.
- Reduced article-content cache TTL default from 7 days to 2.
- Cache now gets purged the moment an article is marked read (client
  sends the article's URL alongside mark-read/mark-all-read so the
  server can delete the matching entry; batched for mark-all-read).
- Google Fonts were loaded via @import inside a JS-rendered <style>
  tag, meaning font loading couldn't start until the entire JS bundle
  downloaded/parsed/executed. Moved to real <link rel=preconnect> +
  <link rel=stylesheet> tags in index.html's <head>, likely explaining
  most of a 4.83s LCP / 0.16 CLS (text repainting/reflowing late once
  the real font finally arrived).
- Fixed sidebar refresh button never showing its spinner: it received
  a `refreshing` prop at the call site but never declared it in its own
  signature, so it was silently dropped.
- Added background content prefetching: after a refresh, the newest
  ~12 unread articles get their full content warmed in the cache (3 at
  a time) so opening one loads instantly instead of fetching on click.

SECURITY
- /api/proxy had no real authentication for its primary use case: the
  auth-gate bypassed requireAuth whenever a request merely looked like
  an iframe navigation (a spoofable header, not a credential) or had
  any token-shaped value at all, without validating it. Removed the
  bypass entirely -- it was also unnecessary, since requireAuth's
  getSessionToken already checks the query ?token= param the client
  always sends for exactly this case.
- cookieJars was a single object shared across every user on the
  process, keyed only by domain -- a login session established by one
  user through the inline browser was silently reused for every other
  user's requests to that domain. Now keyed per userId.
- trust proxy was set to `true` (trust every hop in X-Forwarded-For,
  including attacker-supplied values), confirmed actively firing in
  Vercel's own runtime error logs -- this let anyone bypass IP-keyed
  rate limiting (including on login/2FA) by sending a different fake
  IP per request. Fixed to trust exactly 1 hop on the Vercel-hosted
  server; made configurable (TRUST_PROXY_HOPS, default: trust none) on
  the self-hosted/Electron copy, whose topology isn't knowable here.
- Redirect-based SSRF bypass: checkSsrfSafe only validated the
  hostname a request starts with, but fetchWithCookies used
  redirect:'follow' -- a URL that legitimately resolves to a public IP
  could still redirect to 169.254.169.254 or localhost on its own
  server's say-so, silently followed with no re-validation. Rewrote to
  follow redirects manually, re-checking every hop.
- Hardened the private-IP check against IPv4-mapped IPv6 addresses
  (::ffff:169.254.169.254), which previously matched neither branch.
- Fixed a mutable search_path on a Postgres trigger function flagged
  by Supabase's own security advisor.
- The inline-browser proxy only stripped CSP delivered via the HTTP
  header, missing a <meta http-equiv> CSP entirely (common on Next.js
  sites) -- left in place, a 'self'-based script-src/style-src blocks
  every one of a site's own scripts/stylesheets once this proxy
  rewrites them to absolute, different-origin URLs. Now stripped too.
- Added an SSRF check to /api/article/fetch, which was missing one
  despite fetching a URL that comes from parsed (and possibly
  malicious or compromised) feed content, not user input.

FEATURES
- Folder-level default settings (hide Shorts, inline browser, title
  blocklist, prefer-feed-content, plus the existing thumbnail mode):
  same feed>folder>global precedence throughout, for feeds in a folder
  that haven't set their own override. Deliberately excludes anything
  feed-identity/site-specific (name, URL, CSS/HTML block rules).
- Server-side error logging (error_logs table) plus client-side
  window.onerror/unhandledrejection capture via /api/client-error --
  previously a failed save in a React event handler was an unhandled
  promise rejection with no visibility anywhere.
- Article list thumbnails (large mode, small/square mode, AI-grouped
  clusters) used object-fit:cover, cropping any image that didn't
  match the box's aspect ratio. Switched to object-fit:contain.
- Fixed warped in-article images: a CLS-prevention placeholder
  (aspect-ratio:16/9 on images without width/height attributes -- true
  of nearly all Readability-extracted content) had no object-fit, so
  every non-16:9 image was stretched to fill the box. Added
  object-fit:contain.
- Element-picker rule save was racing a re-fetch that almost always
  won and read stale pre-rule content, made the picker look like it
  silently did nothing; now properly awaited, and a failed save
  surfaces an error instead of leaving the picker stuck with no
  feedback.
